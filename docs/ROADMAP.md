# HR Assist — Roadmap

Документ описывает публичный план развития продукта. Обновляется по мере того как задачи выходят в релиз. Для пользователей — чтобы знать, что впереди; для контрибьюторов — чтобы видеть, где нужна помощь.

## Для кого продукт

HR Assist — это AI-ассистент для **соискателя**. Мы не инструмент для рекрутёров: мы находим подходящие вакансии, объясняем, почему они подходят, помогаем писать отклики и вести воронку.

Чем заменяем на рынке:

- hh.ru Premium (~5 000 ₽/мес) — подсветка откликов, advanced-поиск.
- Карьерных консультантов (10–30 000 ₽ разово) — анализ резюме, сопроводительные письма, объяснение требований.
- AI-агрегаторы уровня getmatch — умный ранжир поверх нескольких источников.

Монетизация, маркетинг и миграции инфраструктуры в этот roadmap **не входят** — это отдельные плоскости, которые мы сознательно не смешиваем с продуктовым планированием.

## Что уже готово

### Phase 0 — Foundation (v0.1.0)

- Жёсткие лимиты дневного бюджета на OpenAI; любой вызов сверх лимита падает в явный отказ, а не в молчаливую ошибку.
- Структурное логирование каждого OpenAI-запроса: модель, токены, стоимость, цель вызова — в одной JSON-строке.
- Rate-limit с `slowapi` на auth-эндпоинтах против брутфорса и самогенерации кодов.
- Размерные ограничения на все пользовательские строки (email, пароль, ФИО, beta-key) — никакой отправки мегабайтов в БД.
- Email-доставка кодов через SMTP для прода и через console-режим для локальной разработки с явным редактированием PII.
- Корректный учёт битых кодировок при парсинге вакансий: не молчим, а считаем `skipped_parse_errors`.

### Phase 1 — Actionability (v0.2.0)

- **Предпочтения при поиске** — формат работы (удалёнка / гибрид / офис / любой), готовность к переезду, родной город, желаемые названия позиций. Первые три — жёсткие фильтры; последний — мягкий boost (+0.10 к score при совпадении).
- **Profile self-check** — после анализа резюме пользователь видит, как мы его поняли (роль, грейд, топ-скиллы), и может исправить до запуска подбора.
- **Объяснение совпадений** — на каждой карточке вакансии: «Ты подходишь» (зелёные галочки) и «Чего не хватает» (янтарные прочерки). Нет больше чёрного ящика.
- **Отмена подбора** — длинный job можно прервать из интерфейса без ожидания конца.
- **Трекер откликов** — Kanban `draft → applied → viewed → interview → offer`, статусы меняются руками, история не теряется.
- **AI-сопроводительное** — кнопка «Сгенерировать сопроводительное» на отклике: система пишет до 250 слов, ссылается на 2+ требования и 2+ факта из резюме, не выдумывает опыт. Один вызов на отклик в сутки.
- **Короткий путь из карточки в отклик** — кнопка «Откликнуться» прямо на карточке совпадения создаёт draft-отклик и открывает редактор сопроводительного.

### Phase 1.7 — Matching quality + multi-profile (v0.3.0)

Phase 1.7 появилась вне основной очереди после того, как dogfood-прогон на реальном пользователе показал два product-level баллера: подбор был слишком шумным для senior-кандидата, а один вектор предпочтений размывал два трека поиска работы (IC / management). Починили обе проблемы до того, как двигаться к retention.

**Track A — Качество подбора:**

- **Скоринг.** Минимальный hard-skill overlap = 1 (если оба описания имеют хотя бы по 3 скилла); штраф −0.15 за несовпадение грейда (senior-резюме + middle-вакансия и наоборот); upgrade title-boost со substring-match на token-overlap.
- **Гигиена индекса.** Double-check на `archived` при матчинге, one-off миграция блокирует старые workingnomads/djinni записи, `status=indexed` и `is_vacancy=true` ушли в Qdrant-filter payload.
- **Time decay предпочтений.** Экспоненциальное затухание веса лайков/дизлайков с периодом полураспада 30 дней (`exp(-days / 30)`). Фича-флаг `PREFERENCE_DECAY_ENABLED` (в рилизе off, включим после недели телеметрии).

**Track B — Multi-profile:**

Лимит — **2 профиля на пользователя**. Воронка откликов общая (один Kanban), но у каждой карточки бейдж, показывающий, под каким профилем был отклик. Лайки и дизлайки изолированы между профилями.

- **Data model.** `resume_id` на `user_vacancy_feedback` (NOT NULL, backfill через активный профиль); ключ preference-вектора в Qdrant — `uuid5(user_id, resume_id, kind)`; проверка лимита на уровне репозитория.
- **API.** `POST /resumes/{id}/activate`, `GET /resumes/active`, `PATCH /resumes/{id}` для лейбла, `POST /vacancies/recommend` скоупится по активному профилю.
- **UI.** Переключатель профилей в topbar, бейджи на карточках Kanban, гейт «удалите один профиль, чтобы загрузить новый» при попытке загрузить 3-й, inline-редактор лейбла с debounced-PATCH.

### Phase 1.8 — Matching relevance hardening (v0.4.0)

Phase 1.8 тоже вышла вне очереди. После релиза `v0.3.0` dogfood вскрыл, что сеньорному IT-резюме в подбор прилетают вакансии из строительства, автопрома и юриспруденции — и даже overlap-floor из 1.7 не спасал, потому что «опыт», «знание», «планирование» как отдельные токены бриджили неродственные требования через bag-of-words.

Починили двумя направлениями сразу, чтобы dogfood-пользователь получил чистую выдачу в одном релизе:

- **Direction A — Domain gate.** Новый фильтр `_has_domain_compatibility` до хард-фильтров: если резюме IT-рутовое, а у вакансии в `domains` ни одного IT-маркера и хотя бы один non-IT-маркер (строительство, автомобили, медицина, юристы, общепит, логистика и ещё ~60 сабкатегорий) — вакансия отсеивается до scoring. Прямое пересечение токенов доменов и ambiguous-случаи проходят, чтобы не ломать кросс-индустриальные IT-переходы.
- **Direction B — Hard/soft token separation.** Резко сузили «мешок» hard-скиллов, по которому считается requirement-matching: берём только `hard_skills` / `skills` / `tools` / `matching_keywords` / `target_role` / `specialization` / `experience.role`. Free-form поля (`soft_skills`, `strengths`, `summary`, `experience.highlights`) перестали поставлять мусор в bag-of-words. Убрали стемминг-fallback `_tokens_semantically_overlap`, который схлопывал несвязанные русские слова на один стем. Подняли `MIN_RELEVANCE_SCORE` 0.55 → 0.60 и `FALLBACK_MIN_RELEVANCE_SCORE` 0.48 → 0.50, чтобы срезать низко-confidence хвост.

Верификация на реальных данных (user 2): мешок hard-skill токенов — 60 кюрированных записей, 0 утечек (`опыт`/`знание`/`проект`/`команд`/`работы`/`управления`/`сильный` отсутствуют); 6 реальных требований из complained-вакансий 17 и 90 (электроника автомобиля, коммерческие сметы, дизайн-проекты) отклонены по всем путям.

## Дальше

## Что сейчас в работе

Активно готовим **Phase 2 — Retention.** После того как подбор стал релевантным и профили изолированы, возвращаемся к вопросу «почему пользователь вернётся сам». Планируемое: сохранённые поиски, email-дайджесты по новым совпадениям, напоминания по долго незакрытым откликам, полный CRUD на предпочтения (сейчас они правятся только через profile card).

**Phase 3 — Value-Add.** Интервью-прэп по конкретной вакансии (типовые вопросы с привязкой к требованиям), gap-анализ карьеры, зарплатные ориентиры по роли + грейду + городу.

**Phase 4 — Frontend refactor.** Разбить `page.tsx` на страницы (`/matches`, `/applications`, `/profile`) и компоненты, добавить мобильный layout.

**Phase 5 — Infra & data.** Устранить N+1 в списках, ограничить большие JSON-поля в API-ответах, вынести background-очередь из in-memory структур в персистентный store.

**Phase 6 — Quality & observability.** Интеграционные тесты на end-to-end сценарии (регистрация → анализ → подбор → отклик), structured-logs assertions, алерты по бюджету и ошибкам OpenAI.

## Как читать этот файл

- ✅ — выкачено в релиз и работает в проде.
- 🔄 — активная разработка.
- ⏳ — запланировано, но ещё не начато.
- ♾ — постоянная фаза без финального «готово».
- `vX.Y.Z` — публичный git-тег, с которым связан релиз фазы.

Каждая галочка `- [x]` = отдельный merged PR. Подробности отдельных PR — в CHANGELOG и в комментариях к релизам на GitHub.
