Recommendations no longer arrive as one flat list. Each match is classified into one of three tracks ‚Äî `match` (—Ç–æ—á–∫–∞, current level), `grow` (–≤—ã—Ä–æ—Å—Ç, one level above), `stretch` (—Å—Ç—Ä–µ–π—á, two+ levels above) ‚Äî and the user sees them as three collapsible sections with per-track market intelligence at the header.

## What's new

### `/` redesigned as a three-track view

- Match section is expanded by default; grow and stretch are collapsed.
- Each section header carries a vacancy counter and one editorial-style gap line: *¬´70% —Ç—Ä–µ–±—É—é—Ç Kafka ‚Äî —É —Ç–µ–±—è –µ—ë –Ω–µ—Ç¬ª*.
- Stretch section gets a CTA ‚Äî *¬´–ü–æ–∫–∞–∑–∞—Ç—å N –≤–∞–∫–∞–Ω—Å–∏–π —Å –º—è–≥–∫–∏–º–∏ —Ç—Ä–µ–±–æ–≤–∞–Ω–∏—è–º–∏¬ª* ‚Äî that filters the section to vacancies whose `must_have_skills - resume_skills` count is ‚â§ 2.
- Visual treatment: 3 px left rule on each section (match = neutral, grow = blue, stretch = amber). Refined editorial ‚Äî typography-led, no icons, no alarm signals on stretch.

### Backend: track classifier (no LLM)

`backend/app/services/track_classifier.py` ‚Äî pure-Python deterministic rule using:
- `vector_score` (already on the candidate),
- `seniority_diff` (`vacancy_seniority - resume_seniority` over a fixed ordinal scale),
- `skills_overlap` (`|resume_skills ‚à© vacancy_must_have_skills| / |vacancy_must_have_skills|`).

```
stretch:  vector ‚â• 0.55 AND (diff ‚â• 2 OR (diff == 1 AND overlap < 0.5))
grow:     vector ‚â• 0.65 AND diff == 1 AND overlap ‚â• 0.5
match:    everything else (sane default ‚Äî matcher already filtered out junk)
```

Wired into `_candidate_to_match_dict` and the score-cache write path. The `track` value is now persisted on `resume_vacancy_scores` so reuse hits don't need to re-classify.

### Backend: track gap analysis

`backend/app/services/track_gap_analysis.py` aggregates the top-5 missing skills per track from `vacancy_profiles.must_have_skills` (the canonical key, fixed in `v0.12.1`). Returns `vacancies_count`, `top_gaps[{skill, fraction, vacancies_with_gap_count}]`, and `softer_subset_count` for each of the three tracks.

Cached in a new `track_gap_analyses` table with TTL 24 h (`computed_at` per `resume_id`).

New endpoint: `GET /api/resumes/{resume_id}/track-gaps` returning `TrackGapAnalysisOut`.

### Telemetry

Four new events surfaced through the new generic sink `POST /api/telemetry/event` (rate-limited 120/min, fire-and-forget):

`applications.track` column added so apply-rate-by-track and per-track funnels (especially the doomed-apply gate metric for `stretch`) can be measured.
